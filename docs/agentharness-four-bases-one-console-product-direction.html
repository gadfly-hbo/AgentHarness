AgentHarness 四库一台产品方向大纲

目标不是同时建设五个并列 demo，而是把 AgentHarness 建成可被 pi-xanthil 消费、可授权、可发布、可商业化的后台能力中台。

核心结论

先建设 Console + Capability Registry + Package / Release / License / Audit 这条平台主干，再用 PLS 做第一条真实业务纵切，优先商业化 DataBase / OntoBase / KnowledgeBase，最后把 MemoryBase 做成跨产品经验沉淀和能力进化闭环。

第一步：先定共同骨架

先补齐一份 Harness Consumption Contract，明确外部产品如何消费 AgentHarness，而不是一开始陷入各库内部表结构。

  • Capability：能力模块，例如 database.pls-profile、ontobase.pls-ontology、knowledgebase.industry-docs。
  • Package：可发布、可安装、可授权的交付单元。
  • Contract：产品消费接口。
  • License：客户是否购买、是否可用、是否降级。
  • Release：版本、变更记录、兼容性。
  • Audit：谁安装、谁调用、谁修改、谁批准。

五个模块的权威边界

DataBase

权威：事实数据、表、view、字段、指标、血缘、数据质量。

OntoBase

权威：业务对象、属性、关系、口径、规则、语义解释。

KnowledgeBase

权威：文档、资料、切片、来源、索引、知识引用。

MemoryBase

权威：经验、偏好、反馈、长期记忆、晋升与遗忘生命周期。

Console

权威：开发、实验、治理、审批、发布、分发、审计。

关键原则：Console 是控制平面，不是四库的宿主。四库各自独立，Console 只通过显式 contract 编排它们。

建设优先级

优先级 建设内容 原因
P0 Console 骨架、Capability Registry、Release / Distribution 没有控制平面，四库会各自长成项目；有控制平面，四库才会变成平台。
P1 DataBase、OntoBase、KnowledgeBase 这三库最容易被客户理解和采购，适合作为企业增强模块。
P2 MemoryBase 先服务内部反馈闭环和跨产品经验沉淀，再逐步变成商业增强能力。

用 pi-xanthil 做第一个 Host

第一期不要抽象到“支持所有产品”，直接以 pi-xanthil 为第一个消费方，打通真实闭环。

四阶段路线

  1. Stage 0: Platform Skeleton：建设 Console + Capability Registry + Package / Release / License / Audit 的最小平台骨架。
  2. Stage 1: PLS Vertical Slice：用现有 PLS 场景贯穿 DataBase + OntoBase + KnowledgeBase + Console，做出一个能被 pi-xanthil 安装和使用的能力包。
  3. Stage 2: Commercial Modules：把 DataBase、OntoBase、KnowledgeBase 做成可单独购买、可组合购买的 enterprise modules。
  4. Stage 3: MemoryBase Closed Loop：把 pi-xanthil 的用户反馈、agent 行为、tool / skill 使用效果回流到 MemoryBase，形成长期优化资产。

pi-xanthil 资产迁移判断

AgentHarness 不应从零建设，而应以 pi-xanthil 已有骨架为母体，做能力治理层外拆。迁移方式不是复制目录，而是把治理权、能力定义、评估发布、行业包和长期资产迁到 AgentHarness；把用户体验、工作区运行、安装启用和前端应用留在 pi-xanthil。

pi-xanthil 资产 建议归属 处理方式
harness-packs/ AgentHarness Packages / CapabilityPacks 最适合直接迁移，作为行业能力包、商业化能力包和安装包标准的母体。
server/tools/registry.ts 与 server/tools/* AgentHarness Console / Tool Registry 迁移 tool manifest、注册制、风险声明、输出契约和评估治理；pi-xanthil 保留运行网关。
skill / tool evaluation、tool-policy、harness-audit AgentHarness Console 这些属于开发、实验、治理、审批、发布、审计能力，不应长期留在产品侧作为权威。
onto-prompts、onto-validator、onto-export、onto-extract OntoBase 抽出本体校验、导出、抽取、source binding 和版本治理；pi-xanthil 保留展示和应用。
knowledge-retrieval.ts KnowledgeBase 复用 tokenizer、BM25、doc/chunk ranking；不要硬搬 workspace DB 绑定。
memory-maintenance、memory-aging、memory-dedup、memory-consolidation MemoryBase 迁移治理、评估、老化、晋升、冲突处理；pi-xanthil 保留 runtime injection。
agentharness-port.ts AgentHarness canonical contract 从 pi-xanthil 反向迁为 AgentHarness 权威 contract；pi-xanthil 保留 client / adapter。

pi-xanthil 保留范围

四库一台独立后，pi-xanthil 应弱化为产品 Host，而不是继续作为四库一台的权威来源。

  • web/src/App.tsx、web/src/tabs/*、ChatPane、BusinessRequirementPane、AnalysisProjectsPane 保留在产品侧。
  • workspace、session、.pi/skills 安装路径和用户前台四模块继续属于 pi-xanthil。
  • /api/extraction-tools/:id/run 可以保留为产品运行入口，但 tool 的权威定义、评估和发布应迁到 AgentHarness。
  • pi-xanthil 后续定位为：产品 Host、工作区 Host、用户交互 Host、AgentHarness capability 的安装与运行 Host。

迁移开工顺序

  1. 先建 AgentHarness Capability Contract，把 pi 现有 agentharness-port.ts 升级成 AgentHarness 权威 contract。
  2. 迁移 harness-packs，形成 AgentHarness 的行业能力包和商业能力包标准。
  3. 迁移 Tool / Skill Registry + Eval，优先建立 Console 的治理价值。
  4. 迁移 OntoBase 能力核，先处理 onto-prompts、onto-validator、onto-export，再处理持久化结构。
  5. 迁移 KnowledgeBase 检索核，抽算法和契约，不硬搬 pi 的 workspace 数据库。
  6. 迁移 MemoryBase 治理核，抽治理、评估、老化、晋升；pi 保留运行时注入。
  7. 让 pi-xanthil 切换为消费方，逐步拿掉本地权威实现。
